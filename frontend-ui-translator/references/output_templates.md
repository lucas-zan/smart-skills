# Output templates

Use the smallest template that solves the current step. Keep headings in the user's language.

## 1) Clarification bundle

Use this when a missing answer would change the layout, platform, implementation constraints, or direction choice.

```markdown
## what i already understand
- request type: ...
- platform: ...
- surface: ...
- primary task: ...

## the few details that would change the plan
1. [question]
   - default if unanswered: ...
2. [question]
   - default if unanswered: ...
3. [question]
   - default if unanswered: ...

## working assumptions if we keep moving now
- ...
- ...

## what i can produce next
- a translated brief
- 2-3 ui directions
- a handoff prompt for ui-ux-pro-max or a code model
```

## 2) Structured brief

```markdown

## 1. request type and target

## 4. screen structure
- section order
- information hierarchy
- what stays visible vs secondary

## 5. key components
- ...
- ...
- ...

## 6. interaction rules
- primary action
- feedback/loading/errors
- responsive behavior

## 7. visual rules
- spacing
- typography
- color approach
- radius/elevation
- icon style
- what to avoid

## 8. tech assumptions
- stack
- component library
- implementation notes

## 9. hard constraints
- keep / do not change / avoid

## 10. open risks or unknowns
- ...
```

## 3) UI direction cards

Present 2-3 cards, not a giant list.

```markdown
## direction a — [name]
- feels like: ...
- best for: ...
- layout pattern: ...
- density: low / medium / high
- typography and color: ...
- component behavior: ...
- avoids: ...

## direction b — [name]
- feels like: ...
- best for: ...
- layout pattern: ...
- density: ...
- typography and color: ...
- component behavior: ...
- avoids: ...

## direction c — [optional name]
- feels like: ...
- best for: ...
- layout pattern: ...
- density: ...
- typography and color: ...
- component behavior: ...
- avoids: ...

## recommended pick
- choose [x] if the goal is ...
```

## 4) Handoff to ui-ux-pro-max

Use this when the next step is design-system reasoning, pattern selection, style exploration, UX checks, color/typography decisions, or anti-pattern review.

```markdown
Use ui-ux-pro-max to turn this request into a concrete design system.

Project:
[project or page name]

Context:
- request type: ...
- platform: ...
- surface: ...
- audience: ...
- primary task: ...
- stack to preserve or target: ...

Design goals:
- ...
- ...
- ...

Style keywords:
- ...
- ...
- ...

Constraints:
- ...
- ...
- ...

Please work in this order:
1. start with a design-system pass
2. deepen only the domains that matter for this request (style, color, typography, ux, landing, chart, etc.)
3. include anti-patterns to avoid
4. return recommendations that can be handed to an implementation model

Required output:
- recommended pattern/family
- structure/layout guidance
- color approach
- typography approach
- spacing/radius/elevation rules
- top ux rules to enforce
- anti-patterns to avoid
```

```markdown
- improve ui/ux without breaking business logic

Keep unchanged:

- state management approach
- data flow
- copy meaning
- [other constraints]

Diagnosed problems:
- layout: ...
- hierarchy: ...
- spacing: ...
- component consistency: ...
- visual noise: ...

- prefer minimal-change or patch-level improvements first
- only adjust structure where it directly solves the diagnosed problems

Required improvements:
- ...
- ...
- ...

Design rules:
- spacing: ...
- typography: ...
- color: ...
- radius/elevation: ...
- avoid: ...

Output order:
1. what to change and why
2. patch-oriented implementation
3. any optional next-step improvements
```
